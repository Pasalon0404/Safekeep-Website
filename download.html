What’s New in v1.2b

A major usability and standards compliance release. Wallet records are now a single portable PDF, descriptor state syncs across the entire vault, and all key exports conform to BIP-32 standard types — eliminating the last friction points in common coordinator workflows.

v1.2b
  • New Wallet Record Overhaul. The wallet record is now generated as a clean, single-page PDF. All descriptor and key information is formatted for easy printing and long-term offline storage.
  • New Deep State Sync. Descriptor, seed hints, passphrase, and the derived m/0/0 address are now synchronized across the vault in a single operation. All tools see consistent wallet state from boot.
  • New Standards Compliance — xpub / tpub. All extended public key exports now use standard BIP-32 xpub and tpub formats, replacing the legacy zpub / SLIP-132 types. Compatible with every major coordinator out of the box.
  • New Dynamic Passphrase Support. Passphrase-protected wallets can now be accessed without re-deriving from scratch. Enter or swap your passphrase at any point during a session and all dependent state updates automatically.
  • Fix Encrypted Notes — Save button lifecycle bug. The Save button in Encrypted Notes was unresponsive after the first use within a session. The SPA lifecycle issue causing this has been resolved; Save now works reliably across any number of note edits.
  • Fix Address derivation regression. A regression introduced by the xpub normalizer caused incorrect receive addresses to be derived in certain derivation path configurations. Derivation is now correct across all supported script types.
  • Fix Layout rendering in strict browsers. Several layout edge cases caused visual breakage under strict content security policies and older WebKit versions. These have been resolved for consistent rendering across all supported environments.

Don’t Trust, Verify

A download that’s been tampered with could steal your seed the moment you type it. Verifying the file takes about two minutes and is the only way to be sure you received exactly what was released. Complete all three steps before writing the image to your USB drive.

Step 01 Import the Developer Key

Download developer-pubkey.asc from the GitHub repository and import it. You only need to do this once.

Linux / macOS / Windows 
# Import the maintainer's public key
$ gpg --import developer-pubkey.asc

GPG is pre-installed on most Linux distributions. macOS users will need to install it, typically via Homebrew (brew install gnupg). Windows users can install Gpg4win.

Step 02 Verify the Manifest Signature

Confirm that SHA256SUMS.txt was signed by the maintainer’s key and has not been modified since signing.

Linux / macOS / Windows 
# Verify the signature over SHA256SUMS.txt
$ gpg --verify SHA256SUMS.txt.asc SHA256SUMS.txt

gpg: Signature made ...
gpg: using RSA key XXXXXXXXXXXXXXXX
gpg: Good signature from "SafeKeepVault Maintainer ..."

You must see Good signature in the output.

Note: You may also see a warning stating “This key is not certified with a trusted signature.” This is standard behavior for newly imported keys. As long as it explicitly says Good signature, the file is verified. Any other result — including “BAD signature” or a key mismatch — means the manifest was tampered with. Do not proceed.

Step 03 Verify the Image Hash

Confirm that the image file you downloaded matches the SHA-256 hash recorded in the verified manifest.

Linux 
# Check image hash against SHA256SUMS.txt
$ sha256sum --check SHA256SUMS.txt
# Expected output: safekeep-os-v1.2b.img: OK
macOS 
# Check image hash against SHA256SUMS.txt
$ shasum -a 256 -c SHA256SUMS.txt
# Expected output: safekeep-os-v1.2b.img: OK

For Linux and macOS, OK confirms the image is byte-for-byte identical to the file the maintainer signed. Any FAILED result means the download is corrupt or was modified — delete it and download again.

Windows (PowerShell) 
# Generate the image hash
PS Get-FileHash safekeep-os-v1.2b.img -Algorithm SHA256

For Windows, open SHA256SUMS.txt in Notepad and visually confirm that the hash generated in your PowerShell window perfectly matches the text file. If they do not match, delete the download immediately.

Flash & Boot

Write the image to a dedicated USB drive (4 GB or larger). The host machine’s internal drive is completely bypassed and remains untouched throughout the session.

Option 1 Graphical Interface (Recommended)

For the most straightforward experience across Windows, macOS, and Linux, we recommend using BalenaEtcher (or Rufus for Windows):

  1. Open the flashing utility and select the extracted safekeep-release-v1.2b.img file.
  2. Select your target USB drive.
  3. Click Flash and wait for the verification to complete.
Option 2 Command Line (Linux / macOS)

Advanced users can utilize the dd utility to write the image directly.

Linux / macOS 
# Flash with dd — replace /dev/sdX with your USB device
$ sudo dd if=safekeep-release-v1.2b.img of=/dev/sdX bs=4M status=progress oflag=sync
Booting the Vault

Once the drive is flashed, configure your computer’s UEFI/BIOS settings to boot from USB. Insert the drive and power on. SafeKeepVault initializes in under 30 seconds—delivering a hardened, distraction-free environment with zero internet connectivity. Just your tools and your keys.
